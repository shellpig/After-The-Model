# 主角公寓

> 第一個可玩場景。此文件只記錄主角公寓場景內的敘事、互動物、驗收方向。

## 場景定位

主角某一天在一間公寓中醒來。

他不知道這裡是哪裡，也不記得自己是誰。當他嘗試開門離開時，發現門被上鎖，而且自己不知道如何打開。

第一個關卡是小型室內探索：

```text
醒來
-> 調查房間
-> 收集奇怪線索
-> 線索逐步組合成身份 / 住處 / 職業資訊
-> 主角意識到這是自己的公寓
-> 找到打開公寓門的方法
-> 離開公寓
```

## 已完成互動（Gate）

### 大門

節點：

```text
DoorExitArea
```

Script：

```text
interactable_area.gd
```

參數：

```text
interaction_id = "door_exit"
prompt_text = "E: 開門"
message_id = "door_locked"
required_knowledge = "identity_door_unlock_method"
```

`interactable_area.gd` 已新增：

```gdscript
@export var required_knowledge := ""
```

位置：

- 公寓右側門前。
- 互動區由使用者手動微調。

靠近提示：

```text
E: 開門
```

未解鎖時按 E：

```text
門上了鎖, 而你發現自己不知道如何打開...
```

Message id：

```text
door_locked
```

`apartment_room.gd` 的 `MESSAGES` 必須新增：

```gdscript
"door_locked": "門上了鎖, 而你發現自己不知道如何打開...",
"door_opened": "你想起來了。門鎖不是壞了, 是你忘了操作方式。"
```

解鎖條件：

```gdscript
GameState.has_knowledge("identity_door_unlock_method")
```

解鎖後行為：

- MVP 不做場景切換。
- 若已取得知識，按 E 顯示：
  ```text
  你想起來了。門鎖不是壞了, 是你忘了操作方式。
  ```
- 後續接 `SceneRouter` 後，再把成功分支改成轉場離開公寓。

派發規格：

```gdscript
match current_interactable.interaction_id:
  "door_exit":
    if GameState.has_knowledge(current_interactable.required_knowledge):
      show_message("door_opened")
    else:
      show_message("door_locked")
```

驗收：

- 未取得 `identity_door_unlock_method` 時，靠近大門顯示 `E: 開門`。
- 未取得知識時按 E，只顯示 `door_locked` 訊息，不切場景、不移動玩家。
- 取得知識後按 E，顯示 `door_opened` 訊息。
- 大門不檢查背包物品，不依賴鑰匙 item。

## Phase 2 解謎設計

> 機制鏈與主線敘事已定稿；標記「待細節討論」的文字/數值留待 Phase 2 細節步驟討論時補。
> 設計原則：線性因果鏈，每一步只解鎖下一步，不開放亂序。呼應主角「對抗遺忘」主題——他是個忘了自己的拾遺者。

### 機制鏈總覽（B0–B9）

```text
B0 醒來（床）
B1 試門（鎖著, 沿用現有 door_locked）
B2 examine 房間 → 兩條線索筆記（善後員 / 拾遺者）
B3 戴上解碼手套（認領身份；手套線索筆記於此由 R 查看/裝備產生）
B4 解碼櫥櫃魔術方塊 → 「解碼方塊」+ 筆記
B5 線索揭示牆上偵測終端
B6 啟動終端 → 進入聲納模式
B7 聲納最強區停留 5 秒 + 鎖定環 → 顯露牆內隱藏插槽
B8 解碼方塊放入插槽（不可取出）→ 電磁聲響 → 提示語音 → add_knowledge(identity_door_unlock_method) → 回憶 + 伏筆
B9 開門（純 has_knowledge gate, 不改現有大門程式）
```

### 子階段對應與 examine 行為（2-A 定稿）

子階段拆分（規格意圖見 `遊戲規格書.md > Phase 規劃 > Phase 2 > 子階段拆分`，實作契約見 `開發設計方針.md`）：

```text
2-A 線索 examine（桌上電腦 / 左牆錄音機 2 條）
2-B 解碼手套 + 魔術方塊→解碼方塊（含手套線索筆記）
2-C 容器擴充（accepted_item / deposit_locked）
2-D 偵測終端 + 聲納 reveal 隱藏插槽
2-E 插槽放入整合 + 全鏈手動驗收
```

**2-A examine 行為定稿**（套用於桌上電腦、左牆錄音機兩個互動物）：

- 沿用 `interactable_area.gd`，新增 examine 分派；多一個 `note_id` export，`apartment_room.gd` 開 `NOTES` dict 存筆記 payload。
- 按 E → MessageBox 顯示「現場敘述句」（`MESSAGES`）＋ `add_knowledge(NOTES[note_id])` 寫入該線索筆記 ＋「已記入筆記」提示。
- 現場敘述句與筆記正文分離維護。
- 兩物**開局即可 examine，無前置 gating**。
- 已記過再 examine：**重播現場敘述句**，不重覆加筆記；若要控制「已記入筆記」提示，使用 `GameState.has_note(note_id)` 判斷筆記是否已存在。
- 手套線索筆記（`clue_gloves_decoder`）**不在 2-A**，改由 2-B 物品側互動產生（見下方手套互動物段）。
- 待 content 補：兩物 prompt 文字、「已記入筆記」字樣。

### 互動物：解碼手套（複用 `fingerless_gloves`）

- 預載已在背包的 `fingerless_gloves` 即此物，Phase 2 賦予解碼功能。
- 行為：**裝備狀態下**，物品 A 會被解碼成物品 B。
- 觸發時機（任一）：
  - 把可解碼物「拿上手」＝從容器移入背包的瞬間。
  - 裝備手套狀態下對該物按 R 查看資訊（防卡關 fallback；modal 要刷新顯示解碼後物品）。
- 未裝備手套時：對可解碼物使用 / 查看都只是普通物品，無任何轉換。
- **不回掃背包**（2-B 定稿）：裝備手套不會自動轉換背包內既有可解碼物；先拿後戴的情況靠 R 查看補解。
- 解碼回饋（2-B 定稿）：解碼當下跳 MessageBox 敘事 + `add_knowledge` 解碼方塊筆記；拿上手路徑以 `item_moved` payload 的 `target_instance_id` 定位剛移入背包的物品並替換 item_id，R 查看路徑以 modal 當前 instance 替換；item_id 變更後不再重複觸發。
- 線索筆記 `clue_gloves_decoder`（分類：線索，寫得隱諱）；**於首次裝備或首次 R 查看手套任一觸發，一次性**：
  ```text
  標題：不只是手套
  這雙手套你戴得很習慣, 習慣到忘了它哪裡不對勁。
  指尖那圈接點碰到某些東西時, 會有反應。
  你還想不起它是用來「讀」什麼的——但你的手記得。
  ```
- 查看訊息（R）：
  ```text
  半截手套, 指節處的布料磨得發亮。右手食指內側有一圈細小的接點, 不像普通手套該有的東西。
  你戴上時, 指尖有極輕微的、像是在「讀取」什麼的震動。
  ```

### 互動物：魔術方塊 →（解碼）→ 解碼方塊

- 新增可解碼物，預載於櫥櫃（`cabinet_storage`）。
- 解碼前：外觀為五顏六色的普通魔術方塊小玩具；未戴手套時使用 / 查看都只是普通玩具。
- 解碼後：變成「**解碼方塊**」，配色轉 cyberpunk；物品名稱「在公寓裡找到的解碼方塊」。
- 解碼當下獲得筆記「解碼方塊」（分類：線索 / 道具，待定）：
  ```text
  標題：解碼方塊
  一種普遍用於解開設備功能的道具, 性質有點像鑰匙。
  放入對應的插槽, 就能開啟特定功能。
  ```
- **伏筆**：解碼方塊是跨關卡的通用道具家族；公寓這顆只是其中一個功能版本，後續關卡會獲得其他功能的方塊。
- 資料模型（2-B 定稿）：`worn_rubiks_cube` `max_stack=1`，metadata 帶 `decodable_to`（指向 `decoder_cube`）+ 所需裝備條件；解碼後 `decoder_cube` 無 `decodable_to`。解碼對應做成資料欄位，不為單一物寫死。
- 物品 id、icon、解碼前後 description 細節：**待細節討論**。

### 互動物：桌上電腦螢幕（善後員線索）

- 新增 examine 互動物。靠近 `E: 查看螢幕`（文字待定）。
- 互動訊息：
  ```text
  螢幕還亮著, 一份新的派工單正自己跳出來, 沒有寄件人。
  ```
- 筆記 `work_ai_cleanup_role`（分類：工作；沿用現有 id，body 改為下文）：
  ```text
  標題：AI 善後員
  派工單一筆一筆自己跳出來, 地址、編號, 註記欄寫著「殘留清除」「記憶體焚毀」。
  從沒見過發派的人, 只有螢幕那頭簡短的指示, 從不寒暄, 也從不出錯。
  原來你靠這個過活——收拾 AI 留下的、人們不想再看見的東西。
  ```
- **埋伏筆**：發派方刻意寫得非人、像系統（「自己跳出來」「從沒見過發派的人」「從不寒暄, 從不出錯」），為後續揭曉「老闆其實是 AI」鋪路。此刻玩家只當成冷漠的外包系統。

### 互動物：左牆錄音機 / 老歌錄音帶（拾遺者線索）

- 新增 examine 互動物，位於場景偏左牆面。靠近 `E: 按下播放`（文字待定）。
- 互動訊息：
  ```text
  錄音機裡卡著一捲帶子。按下播放, 是首沒人記得的老歌, 雜訊裡有人輕輕跟著哼。
  ```
- 筆記 `identity_gleaner`（分類：身份）：
  ```text
  標題：拾遺者
  牆上整排都是舊帶子, 老歌、舊廣播、不知道誰的留言。
  這些早該被善後員銷毀的東西, 你卻一捲一捲留了下來。
  你一邊清除過去, 一邊偷偷把它撿回家。
  ```
- 設計意圖：「善後員（抹除）」與「拾遺者（收藏）」互相矛盾，正中角色雙重身份與「對抗遺忘」。

### 互動物：偵測終端（聲納開關）

- 新增牆上固定互動物。世界觀設定：**主角當年自己裝的舊偵測終端**，掃的是他自己藏在牆裡的插槽。
- examine → `E: 啟動`（文字待定）。
- 啟動後全房進入**聲納模式**：
  - 玩家在房間水平移動時發出嗶聲，**越靠近隱藏插槽，頻率/強度越高**（純 1D 水平距離）。
  - 為 accessibility 同步提供視覺強度條（meter），不只靠音效。
  - 在最強區**固定停留 5 秒**：顯示鎖定環 +「保持不動…」提示；中途離開最強區則重置。
  - 鎖定完成 → 顯露牆內隱藏插槽（隱藏互動物 reveal）。
- 因果咬合：他藏了東西 + 留了找回它的工具 → 卻忘了。補上「主角當初為何藏插槽」的著力點（刻意藏、自備找回手段）。
- 聲納頻率曲線、最強區寬度、音效素材：**待細節討論**。

### 互動物：隱藏插槽（鎖定容器）

- reveal 後成為可互動容器，**沿用一般雙欄容器 UI**（背包左 / 插槽右），不特化單格——插槽容器格數可變，公寓這個用 1 格即可。
- 規格（2-C 定稿）：
  - 用 `configure_container(container_id, slot_count, accepted_item, deposit_locked)` 設定；公寓：`accepted_item = ["decoder_cube"]`、`deposit_locked = true`、1 格。
  - `accepted_item` 為清單；放入清單外物 → 「放不進去」（沿用容器白名單拒收）。
  - `deposit_locked`：放入後不可取出（見規格書 External Container API > 容器選項擴充）。
  - UI 透過 `GameState.get_container_config(container_id)` 讀取白名單與鎖定狀態，用於區分「放不下了。」與「放不進去」等訊息。
  - 取出被擋的訊息字樣：**待 2-E content 定**。
- 放入解碼方塊後流程：
  1. 電磁聲響（音效，待定素材）。
  2. MessageBox 提示語音（暫用文字表現）：
     - 告知「房門已解鎖」。
     - 主角回想整段記憶：他**自己做了某種機關鎖住房門**。

## UI 行為要求

公寓場景應遵循全遊戲通用 UI 規格：

- 世界模式：主角可走路，E 與最近互動物互動。
- 大型 UI 開啟時：主角不能移動。
- Prompt 永遠跟隨主角上方。
- MessageBox 自動適配文字尺寸並逐字顯示。
- 容器互動最終要改成雙欄：
  ```text
  左：背包
  右：容器
  ```
