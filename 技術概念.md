# After-The-Model 技術概念

> 目的：整理目前已討論的技術方向，供其他 AI 或 reviewer 做技術審核。

## 目前開發目標

先做一個本機 PC 可玩的 MVP，不急著處理 Steam / iOS 正式上架流程。

MVP 範圍暫定：

- 一條街
- 一個地鐵站
- 一個小公寓
- 2 個 NPC
- 1 個零工任務
- 使用 placeholder / AI 生成素材建立可玩流程

第一階段重點不是商業級完成品，而是驗證：

- 2D 橫向探索是否成立
- 場景切換是否順
- NPC 對話與任務流程是否可讀、可玩
- 地鐵作為區域移動核心是否適合
- 近未來低階層 cyberpunk 的氛圍是否能被快速建立

## 建議引擎

建議使用 Godot 4.x，語言以 GDScript 為主。

理由：

- 遊戲目標是 2D 橫向探索，Godot 的 2D workflow 適合。
- Godot 專案可跨 Windows / macOS / iOS / Android / Linux。
- GDScript 對 MVP 迭代速度快，避免 C# 在 iOS 匯出上的額外風險。
- 對小型獨立遊戲來說，Godot 成本低、安裝與版本管理相對單純。

目前討論時建議版本：

- Godot 4.6.3 stable
- Godot export templates

備註：版本建議是依 2026-05-25 查到的穩定版狀態。正式開發前仍應再確認 Godot 最新 stable 與 export templates 是否一致。

## 本機 PC MVP 需要安裝的項目

先做到本機 PC 可玩的 MVP，只需要以下四項：

1. Git
2. Godot 4.6.3 stable + export templates
3. Python venv + Pillow + numpy
4. agent-sprite-forge skills

建議安裝順序：

```text
Git
-> Godot 4.6.3 + export templates
-> Python venv + Pillow + numpy
-> agent-sprite-forge skills
-> 開始做 PC MVP
```

Steamworks、Xcode、Apple Developer Program、App Store Connect 都可以等 PC MVP 能跑、玩法確定後再處理。

## Python 與 venv

本專案規則要求使用專案 venv：

```text
.\.venv\Scripts\python.exe
```

用途：

- 安裝 agent-sprite-forge 後處理需要的套件
- 執行素材後處理腳本
- 未來執行輔助工具或測試腳本

agent-sprite-forge 目前需要的 Python 套件：

- Pillow
- numpy

## agent-sprite-forge 使用方式

預計使用 `agent-sprite-forge` 協助 MVP 圖像素材。

適合用途：

- 主角 sprite sheet
- NPC sprite sheet
- 場景物件 props
- 地鐵站 / 街道 / 小公寓的概念素材
- 透明 PNG / GIF
- prop pack 切圖
- 簡單分層地圖或 Godot 可用素材

它主要提供兩個 Codex skills：

- `$generate2dsprite`
- `$generate2dmap`

使用定位：

- 適合快速建立可玩的視覺原型。
- 適合在 MVP 階段大量產出草稿素材。
- 不應直接視為最終美術品質保證。

主要風險：

- AI 生成素材容易風格不一致。
- 同一角色不同動畫可能比例、服裝、輪廓不穩。
- 場景光影與色盤可能每批不同。
- 後期仍需要人工用 Aseprite / Krita 修圖與統一風格。

建議補充一份小型 art bible：

- 角色比例
- 色盤
- 線條粗細
- 陰影規則
- 富人區 / 貧民區視覺差異
- 霓虹、雨夜、CRT、地下道等關鍵視覺元素規格

## agent-sprite-forge 安裝位置

不建議把 `agent-sprite-forge` repo 放進 `AfterTheModel` 遊戲專案內。

建議放在專案外的工具資料夾，例如：

```text
C:\_work\AI_Work\Tools\agent-sprite-forge
```

原因：

- 避免外部工具 repo 混入遊戲專案。
- 避免未來 git commit 誤收外部工具內容。
- 可讓其他遊戲專案重複使用同一份工具。

安裝方式概念：

```text
clone agent-sprite-forge 到專案外
-> 檢查 skills/generate2dsprite 與 skills/generate2dmap
-> 用 .\.venv\Scripts\python.exe 安裝 Pillow / numpy
-> 複製 skills 到 C:\Users\User\.codex\skills
-> 重新開 Codex session 讓 skills 載入
```

## Placeholder 定義

Placeholder 是暫時佔位素材。

MVP 階段先用簡單、可替換的素材把遊戲流程跑起來，避免先投入大量正式美術後才發現玩法要重改。

例子：

| 正式素材 | Placeholder 版本 |
|---|---|
| 主角完整動畫 | 簡單人形或 AI 生成粗版 sprite |
| 地鐵站精緻背景 | 簡化月台、站牌、入口 |
| NPC 立繪 | 不同輪廓或色彩的角色草稿 |
| 小公寓 | 床、桌、窗、門等基本物件 |
| 任務 UI | 純文字提示與簡單互動按鈕 |

## Steam 發佈路線

Steam 可放在 PC MVP 之後。

需要準備：

- Steamworks Partner 帳號
- Steam Direct fee，單一 app 目前為 100 USD
- Steamworks SDK
- SteamPipe / ContentBuilder
- Steam client 測試
- Godot Windows export

GodotSteam plugin 是可選項。

不需要一開始就接：

- Steam achievements
- Steam cloud saves
- Steam overlay
- Steam stats
- multiplayer / lobby

MVP 建議先做到：

- Windows build 可執行
- 可透過 Steam 測試 branch 安裝與啟動
- 不依賴 Steam API 也能完整跑完 MVP

## iOS 發佈路線

使用者有 MacBook 與 iPhone，因此手機平台可優先考慮 iOS。

iOS 需要準備：

- MacBook
- Xcode
- Apple Account
- Godot iOS export templates
- Apple Developer Program
- App Store Connect
- TestFlight

注意：

- Godot iOS export 必須在 macOS + Xcode 上完成。
- 正式 TestFlight / App Store 發佈需要 Apple Developer Program。
- Apple Developer Program 目前為 99 USD/year。

iOS 不是單純多按一次 export，還需要額外處理：

- 觸控移動或虛擬搖桿
- 互動按鈕位置
- 對話框字體大小
- safe area
- iPhone 長螢幕比例
- iPad 比例是否支援
- 存檔位置
- 觸控版任務 UI

建議流程：

```text
PC MVP 可玩
-> 匯出 Windows build
-> MacBook 匯出 iOS Xcode project
-> iPhone 實機測試
-> TestFlight 小範圍測試
-> 再考慮 App Store 正式送審
```

## 時間粗估

只做 PC 可玩 MVP：

- 最快：1-2 週
- 較合理：3-4 週
- 若追求強氛圍與較完整美術：5-8 週

若同時納入 iOS 實機與觸控 UI：

- 建議估 4-6 週起跳

時間差主要來自：

- 美術一致性
- 場景細節
- 任務流程完整度
- UI / 對話 / 音效 polish
- iOS signing、Xcode、TestFlight 與觸控操作調整

## 初期不建議做的事

MVP 階段先不要做：

- Steam achievements
- Steam cloud saves
- 手機 IAP
- 多平台同步存檔
- 複雜背包系統
- 大型任務系統
- 大量隨機事件
- 多語言完整支援
- 複雜 NPC 日程
- 高度抽象的工具框架

理由：這些都可能拖慢第一個可玩版本。先驗證核心體驗比較重要。

## MVP 架構決策

以下為與 reviewer 討論後已確認的 MVP 架構方向，後續實作須遵循。

### 螢幕方向

- 強制 landscape（橫向）
- PC 視窗預設 16:9，手機強制 19.5:9 landscape
- `project.godot` 建議設定 stretch mode = `canvas_items`、aspect = `expand`

### 輸入抽象層

- 所有玩家輸入透過 Godot InputMap 定義抽象 action
- 程式碼只 listen action，不 listen 實體按鍵
- PC 與手機綁不同實體輸入到同一 action

| Action | PC | 手機 |
|---|---|---|
| move_left / right / up / down | WASD / 方向鍵 | 左側虛擬搖桿 |
| interact | E 鍵 / 滑鼠左鍵 | 右側互動按鈕 |
| open_menu | ESC / Tab | 右上選單按鈕 |

啟用 Godot 內建設定簡化跨平台事件:

- `input_devices/pointing/emulate_mouse_from_touch = true`
- `input_devices/pointing/emulate_touch_from_mouse = true`

啟用後 UI 點擊只要寫一次 handler，PC 與手機共用。

### UI 設計規範

- 所有 UI 使用 Control 節點 + Anchor + Container 自適應
- 觸控目標最小尺寸 ≥ 44×44 px（iOS HIG）
- 不依賴 hover 顯示關鍵資訊（手機沒有 hover）
- Tooltip 改成 long-press 觸發
- iOS safe area 用 SafeAreaContainer 處理瀏海與底部 home indicator

### 互動系統

- 單一主互動鍵（PC: E 鍵 / 手機: 右側「互動」按鈕）
- 玩家進入物件互動範圍 → 物件上方顯示 hint
- 按互動鍵 → 彈出 context menu（list 或 radial）
- 多種互動（買 / 對話 / 偷竊 / 檢視）皆為 menu 項目
- PC 與手機 UI 一致

### 敘事資料格式

- 使用 Godot Resource（`.tres`）儲存對話、任務、NPC 資料
- 不寫死在 GDScript 內
- 結構含 i18n key，即使 MVP 只填中文也預留英文欄位
- 純文字格式 git diff 友善

範例結構：

```text
{
  id: "npc_delivery_01_intro",
  speaker: "delivery_guy",
  text_zh: "...",
  text_en: "",
  choices: [ ... ]
}
```

MVP 階段先用純 Resource 結構，若敘事複雜度上升再考慮遷移至 Ink。

### 全域狀態：Autoload Singleton 拆分

避免 God Object，一開始就拆五個 Autoload：

| Autoload | 職責 |
|---|---|
| GameState | 玩家狀態（金錢、HP、目前場景、buff） |
| QuestManager | 任務進度與旗標 |
| DialogueState | 已看過的對話、NPC 互動歷史 |
| SceneRouter | 場景切換、地鐵移動 |
| SaveSystem | 序列化邏輯（讀寫上面四個） |

序列化使用 Godot 原生 `ResourceSaver.save()`，避免手寫 JSON。

### 遊戲時鐘

- 時間不是即時時鐘，而是時段 enum
- TimeOfDay：MORNING / AFTERNOON / EVENING / LATE_NIGHT
- 每個場景在不同時段有不同：
  - 背景光照 / 顏色濾鏡（CanvasModulate）
  - NPC 配置
  - 可互動物件（部分店家深夜關門）

推進規則：

- 睡覺 → 跳到下個 MORNING
- 特定任務事件 → 跳到指定時段
- 地鐵末班車 → LATE_NIGHT（玩家可選擇是否搭）
- 街上漫遊不會自動推進時間

### 存檔策略

- MVP 階段：隨時手動存檔 + 自動存檔（場景切換、重要事件）
- SaveSystem 預留 `can_save_here: bool` 參數
- 未來要改成「只能在安全點存」只需切 flag，不需重寫架構
- 存檔格式使用 `ResourceSaver.save()` 寫入 `user://save.tres`

### Debug 工具

MVP 階段同步建立 debug overlay：

- F1 開關
- 跳場景、給錢、完成任務、跳時段
- 顯示目前 GameState 內容
- 強制觸發特定事件

避免每次測試從頭跑流程。

### Art Bible 時機

- 在生第 5 張 sprite 之前必須完成 art bible
- 否則風格漂移後回不去
- 內容規格已列於本文「agent-sprite-forge 使用方式」段落

## 建議 reviewer 審核重點

請 reviewer 優先看：

- Godot 4.6.3 是否仍是合適的 stable 版本
- GDScript 是否足以支撐後續系統
- agent-sprite-forge 是否適合此類橫向 cyberpunk MVP
- Windows PC MVP -> Steam -> iOS 的順序是否合理
- iOS 觸控 UI 的工作量是否被低估
- ✅ 是否需要更早建立 art bible（已決定：生第 5 張 sprite 前完成）
- ✅ 是否需要先建立任務/對話資料格式（已決定：Godot Resource + i18n key）
- ✅ 是否需要在 MVP 初期就規劃存檔（已決定：隨時手動存 + 預留 can_save_here flag）
