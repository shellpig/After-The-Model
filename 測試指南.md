# 測試指南

> 測試流程與手動驗收步驟清單。
> **撰寫時機**：各子階段的測試項目於**該階段規格定稿時即寫入**（與規格同期，acceptance-test 先行），供實作時據以建測試與自我驗收——不延到驗證時才補。
> **角色歸屬**：測試項目於設計定稿時寫入；之後由 verifier 維護與執行。實作者照清單建測試、不得放寬清單，不改本檔（只列建議）。

## 本檔範圍與邊界

- **規格書「驗收」= 意圖層**：什麼狀態必須為真（what must be true），不寫操作步驟。
- **本檔 = 操作層**：開哪個場景、按哪顆鍵、看什麼畫面（click-by-click）。
- 同一條斷言不抄兩份；本檔的每個清單項用 link 指回規格書對應「### 驗收」段。
- 規格定稿時，測試項目可能引用尚未定的 content（如 prompt 文字）；以「顯示其 prompt」等中性描述先寫，實作時填實際字樣。

#### 2-A 線索 examine（桌上電腦 / 左牆錄音機）

對應規格：`遊戲規格書.md > Phase 2 > 2-A 驗收意圖`。前置：新遊戲進入公寓場景。

桌上電腦（善後員 / `work_ai_cleanup_role`，分類：工作）：
- [ ] 新遊戲剛進公寓、尚未 examine 桌上電腦前，筆記 UI 不應已有 `work_ai_cleanup_role`（Phase 1 demo preload 已移除或改由 examine 覆寫）。
- [ ] 走近桌上電腦 → 顯示其 prompt。
- [ ] 按 E → MessageBox 顯示現場敘述句，且訊息附「已記入筆記」提示。
- [ ] 關閉訊息後開筆記 UI（`J`）→「工作」分類出現 `work_ai_cleanup_role`，標題 / 內容正確。
- [ ] 再走近按 E → 重播同一現場敘述句；回 `J` 確認該筆記仍只有 1 筆（未重覆新增）。

左牆錄音機（拾遺者 / `identity_gleaner`，分類：身份）：
- [ ] 走近左牆錄音機 → 顯示其 prompt。
- [ ] 按 E → MessageBox 現場敘述句 +「已記入筆記」；`J`「身份」分類出現 `identity_gleaner`，內容正確。
- [ ] 重複 examine → 重播訊息、不重覆加筆記。

通則：
- [ ] 開局未做任何前置即可 examine 兩物（無 `required_knowledge` 阻擋）。
- [ ] examine 跳 MessageBox 期間遵守 UI Mode（玩家不能移動、Prompt 隱藏等，依既有規格）。
- [ ] 兩物的現場敘述句與筆記正文為不同文字（敘述句較短、筆記較完整）。

> 待 content 定稿後填實際 prompt 文字與「已記入筆記」字樣。

#### 2-B 解碼手套 + 魔術方塊轉換

對應規格：`遊戲規格書.md > Phase 2 > 2-B 驗收意圖`。前置：新遊戲進入公寓，背包有 `fingerless_gloves`，櫥櫃預載 `worn_rubiks_cube`。

手套線索筆記：
- [ ] 首次裝備手套（E）→ 得 `clue_gloves_decoder` 筆記（分類：線索）+「已記入筆記」提示。
- [ ] 卸下再裝備 / 之後再 R 查看 → 不重覆新增該筆記。
- [ ] （另一存檔）改先 R 查看手套 → 同樣首次即得 `clue_gloves_decoder`。

未裝備手套（不轉換）：
- [ ] 未戴手套，把 `worn_rubiks_cube` 從櫥櫃拿進背包 → 仍是普通魔術方塊，無轉換、無筆記。
- [ ] 未戴手套，R 查看 `worn_rubiks_cube` → 普通玩具描述，無轉換。

裝備手套 + 拿上手觸發：
- [ ] 戴手套後，把 `worn_rubiks_cube` 從櫥櫃移入背包 → 跳 MessageBox 敘事；該格變成 `decoder_cube`（名稱「在公寓裡找到的解碼方塊」）；得「解碼方塊」筆記。
- [ ] headless / debug 驗證：上一步轉換使用該次 `item_moved` payload 的 `target_instance_id` 定位，不依賴背包排序後猜格；背包內不會同時多出 `worn_rubiks_cube` 與 `decoder_cube` 兩個實例。
- [ ] 轉換後該物再搬動（背包↔櫥櫃）→ 不再跳訊息、不重覆加筆記。

裝備手套 + R 查看 fallback：
- [ ]（另一存檔）先未戴手套把 `worn_rubiks_cube` 拿進背包 → 不轉換；之後戴手套，**不自動回掃**（仍是 `worn_rubiks_cube`）。
- [ ] 接上：戴手套狀態 R 查看該 `worn_rubiks_cube` → 跳 MessageBox；modal 刷新顯示 `decoder_cube`；得「解碼方塊」筆記。

通則：
- [ ] 解碼 MessageBox 期間遵守 UI Mode。
- [ ] `worn_rubiks_cube` / `decoder_cube` 皆 `max_stack=1`。
- [ ] `clue_gloves_decoder` 去重使用 note 存在判斷（`has_note` 或等價），不是 `has_knowledge`；線索分類不應被當成大門 gate knowledge。

> 待 content 定稿後填實際 item id、description、MessageBox 敘事與「已記入筆記」字樣。

#### 2-C 容器白名單 / deposit_locked（headless）

對應規格：`遊戲規格書.md > Phase 2 > 2-C 驗收意圖` + `External Container API > 容器選項擴充`。可用 `tests/manual/verify_game_state.gd` 風格的腳本驗證（純資料層，無需 UI）。

向後相容：
- [ ] `configure_container("cabinet_storage", 30)`（省略新參數）→ 任意 item 可放入、可取出，與 Phase 1-D 一致。

白名單（accepted_item）：
- [ ] `configure_container("test_slot", 1, ["decoder_cube"])` 後，`move_one_item_to("test_slot", <decoder_cube instance>)` → `true`，物入槽.
- [ ] 同容器放入非清單物（如 `<canned_food instance>`）→ `false`，來源與容器內容皆不變。
- [ ] 白名單給多個 id（如 `["decoder_cube","worn_rubiks_cube"]`）→ 兩者皆可放入。
- [ ] `get_container_config("test_slot")` 回傳 `accepted_item = ["decoder_cube"]`、`deposit_locked = false`，且外部修改回傳值不會改 GameState 內部設定。

鎖定（deposit_locked）：
- [ ] `configure_container("test_locked", 1, ["decoder_cube"], true)` 並放入 decoder_cube 後，`move_one_item_to("player_inventory", <該 instance>)`（從鎖定容器取出）→ `false`，容器內容不變。
- [ ] 對 `deposit_locked` 容器**放入**（作為目標）仍正常 → `true`。
- [ ] `get_container_config("test_locked")` 回傳 `deposit_locked = true`，供 UI 區分取出被擋與容量不足。

通則：
- [ ] `move_one_item_to` 全程只回傳 bool，不回原因碼。
- [ ] 成功搬運時 emit `item_moved(move)`，payload 含 `source_container_id`、`target_container_id`、`source_instance_id`、`target_instance_id`、`item_id`。
- [ ] 失敗（任一 false）時來源與目標內容皆無變動。

> headless 執行限制見本檔「Godot 測試環境」段（escalated 權限）。

#### 2-D 投影時鐘 + 聲納 reveal 隱藏插槽

線索麵包屑（營養棒，雙路徑同效）：
- [ ] 開局投影時鐘無 prompt / 不可互動（線索未取得）。
- [ ] 開冰箱 → R 查看 `nutrition_bar_synth_blueberry` → 跳 MessageBox 空包裝藏字敘述；關閉後該道具消失（消耗）、得線索。
- [ ]（另一存檔）把營養棒移入背包 → 單獨背包 footer 顯示 `E: 使用` → 按 E → 同上 MessageBox + 消耗 + 得線索。
- [ ] 營養棒 icon 使用 `res://assets/generated/sprites/items/synthetic_blueberry_nutrition_bar/icon.png`。
- [ ] `J` 線索分類出現 `clue_projection_clock`，標題 / 內容正確。
- [ ] 取得線索後投影時鐘變可互動（顯示 `E: 啟動`）。

背包 footer E 段（Phase 1-E 連帶修正）：
- [ ] 選中可裝備物（如 `fingerless_gloves`）→ footer 顯示 `E: 裝備`；裝備後顯示 `E: 卸下`。
- [ ] 選中 consumable（如 `canned_food`）→ footer 顯示 `E: 使用`（非「裝備/卸下」）。
- [ ] 選中無動作物（如 `old_work_badge`）→ footer **不顯示 E 段**（仍有 R / T / Esc·I）。

- [ ] 雙欄左欄（背包欄）選物 → E 仍為「移動」，不出現「使用」。

聲納啟動 + reveal：
- [ ] 互動投影時鐘 → 進入聲納模式，10 秒倒數開始，出現視覺強度條。

- [ ] 停留最強區帶寬內（目標 X 座標 `±36px`，小幅移動仍算）→ 鎖定環累積 4 秒至滿 → 播放 `res://assets/sound/hidden_slot_reveal.wav` 並 reveal 隱藏插槽。
- [ ] 鎖定中離開最強區帶寬 → 鎖定環重置。

限時與凍結：
- [ ] 啟動後放任 10 秒倒數歸零（不去鎖定）→ 聲納結束、無 reveal；再互動投影時鐘可重新啟動。
- [ ] 聲納進行中按 `I` / `J` 開大 UI → 聲納、倒數、鎖定進度凍結；關閉 UI → 由凍結點接續（未歸零）。

reveal 後：
- [ ] reveal 後隱藏插槽常駐，可如一般雙欄容器開啟。
- [ ] 投影時鐘不再可互動（互動移除）；聲納無法再啟動。

通則：
- [ ] 聲納 / MessageBox 期間遵守 UI Mode（玩家移動、Prompt 隱藏依既有規格）。
- [ ] 新存檔重跑可重現上述全鏈。

> 短音播放頻率曲線、鎖定環視覺、文字字樣待 content 定稿後對齊。

#### 2-G 開場獨白序列

對應規格：`遊戲規格書.md > Phase 2 > 2-G 驗收意圖`；文案見 `subdocs/地點/主角公寓.md > 開場序列（B0 開場獨白，2-G）`；契約見 `開發設計方針.md > 2-G`。前置：新遊戲進入公寓場景。

進場狀態：
- [ ] 新遊戲一進公寓即自動播開場：玩家為趴睡姿（`prone`）、無法移動、互動 prompt 不顯示、MessageBox 顯示第 1 頁。
- [ ] 開場期間按 `I` / `J` / `E` / 移動鍵皆無效（世界互動與大型 UI 全擋）；只有換頁鍵與隱藏 T 有作用。

逐頁播放 / 不可跳過：
- [ ] 每頁文字逐字顯示（約 6 字/秒）；文字未跑完前按 WASDE / 方向鍵 / Esc 皆**無法換頁、不快進、不跳過**。
- [ ] 每頁標題（感官甦醒 / 空白的記憶 / 自囚與尋回）顯示在該頁正文第一行，並隨打字機一起逐字出現。
- [ ] 某頁文字完整顯示後，淡入換頁提示：第 1~2 頁「▼ 繼續」、第 3 頁「▼ 開始」；文字未跑完不顯示提示。
- [ ] 文字跑完後按 WASDE / 方向鍵 / Esc 任一 → 進下一頁。
- [ ] 共 3 頁，依序 P1→P2→P3，每頁文案與 subdoc 全文一致。

起身動畫（第 3 頁）：
- [ ] 第 3 頁一出現即播放 `get_up`（約 1 秒、10 frame、不循環），播完停最後一格並切 `idle`。
- [ ] 起身動畫播放期間第 3 頁文字同時繼續跑（文字不等動畫、動畫不等文字）。
- [ ] 起身（~1 秒）早於第 3 頁文字結束；動畫結束後角色維持站立 `idle`、文字續跑。

結束 / 解鎖：
- [ ] 第 3 頁文字跑完、按 WASDE / 方向鍵 / Esc → MessageBox 關閉、解除輸入封鎖、玩家可自由移動。
- [ ] 開場結束後角色為站立 `idle`（非 prone）；可正常走動、開背包 / 筆記、與互動物互動（接 B1 試門等）。

隱藏測試捷徑（dev skip）：
- [ ] 任一頁 1 秒內連按 3 下 T → 立即補完當頁文字並跳下一頁。
- [ ] 第 3 頁 1 秒內連按 3 下 T → 直接關閉開場、解鎖操作；角色 snap 為站立 `idle`（即使 get_up 未播完）。
- [ ] 連按間隔超過 1 秒（非連續）→ 計數歸零、不觸發跳頁。
- [ ] 換頁提示中不出現任何 T 捷徑字樣（隱藏性）。

每次進場：
- [ ] 重開新遊戲 / 重進公寓場景 → 開場每次都播（MVP 無「只播一次」flag）。

通則：
- [ ] 開場期間遵守輸入封鎖，玩家不能移動、Prompt 隱藏。
- [ ] 文案以 subdoc 全文為準，逐頁標點一致。

### Phase 3 — 公寓觸控化（手機可玩）
> 驗收意圖見 `遊戲規格書.md > Phase 規劃 > Phase 3`；實作契約見 `開發設計方針.md > Phase 3`。
