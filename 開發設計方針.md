## Phase 2 — 公寓解謎（實作契約）

> 機制鏈與驗收意圖見 `遊戲規格書.md > Phase 規劃 > Phase 2`；完整流程與文字見 `subdocs/地點/主角公寓.md > Phase 2 解謎設計`。

### 2-A 線索 examine 互動物

範圍：兩個世界 examine 互動物——桌上電腦（`work_ai_cleanup_role`）、左牆錄音機（`identity_gleaner`）。手套線索（`clue_gloves_decoder`）不在此，見 2-B。

互動物形態（沿用 `interactable_area.gd`，不新增節點類型）：

- `interactable_area.gd` 新增 export：`note_id`（examine 分支用；非 examine 物留空）。
- `apartment_room.gd` 新增 `NOTES` dict（類比現有 `MESSAGES`），key = `note_id`，value = `add_knowledge` 所需的筆記 payload（id / category / title / body / status）。
- `apartment_room.gd` 的派發 `match` 新增 examine 分支：
  - `show_message(message_id)` 顯示現場敘述句（沿用 `MESSAGES`）。
  - `GameState.add_knowledge(NOTES[note_id])` 寫入筆記。
- 現場敘述句（`MESSAGES`）與筆記正文（`NOTES`）分離維護。

行為契約：

- 兩物開局即可 examine，無 `required_knowledge` 前置。
- 重複 examine：每次重播 `show_message`；`add_knowledge` 依其 idempotent 特性不重覆加。若 examine 分支需要控制「已記入筆記」提示是否出現，先用 `GameState.has_note(note_id)` 判斷，不用 `has_knowledge()`。
- 線索筆記文字見 `subdocs/地點/主角公寓.md > Phase 2 解謎設計`。
- Phase 2 開工時，`work_ai_cleanup_role` 不再作為新遊戲預載 demo 筆記；改由桌上電腦 examine 首次寫入，避免 2-A 首次取得驗收被預載污染。

待 content 補：兩物 prompt 文字（`E: 查看螢幕` / `E: 按下播放`）、「已記入筆記」提示字樣。

### 2-B 解碼手套 + 魔術方塊道具

物品與 metadata：

- `worn_rubiks_cube`（魔術方塊，普通玩具）`max_stack=1`；新增 metadata 欄位 `decodable_to = "decoder_cube"` + 所需裝備條件（裝備某具解碼能力的物，公寓即 `fingerless_gloves`）。
- `decoder_cube`（解碼方塊）`max_stack=1`；無 `decodable_to`，搬動不再觸發。
- 解碼能力標在裝備物上（如 `fingerless_gloves` 加一個「具解碼能力」旗標），讓「裝備條件」可組合判斷，不寫死特定手套 id。
- 兩物 id 命名 / icon / description 為 content，定稿前以上列暫名。

轉換觸發接線（兩處，行為一致）：

- **拿上手**：`move_one_item_to("player_inventory", instance_id)` 成功後，controller 讀取該次 `item_moved` payload 的 `target_instance_id`；若目標物有 `decodable_to`、且玩家已裝備滿足條件的裝備 → 更新該 target instance 的 `item_id` 為 `decodable_to`、跳 MessageBox 敘事、`add_knowledge` 解碼方塊筆記。
- **R 查看（fallback）**：ItemDetailModal 開啟中、查看物有 `decodable_to`、裝備條件滿足 → 更新 modal 當前 instance 的 `item_id`；MessageBox 關閉後 modal 刷新顯示 `decoder_cube`。MessageBox 疊在 detail modal 上方期間 detail modal 凍結、不吃鍵盤（capture 暫交 MessageBox），關閉後還原 capture 並刷新（消耗物則連 modal 一起關回 grid）。
- 轉換定位：不得靠背包排序後的格位猜測剛移入物；拿上手路徑一律使用 `item_moved` payload 的 `target_instance_id`。轉換後 UI 上必須表現為「剛操作的那個物品變了」，不得另新增第二個方塊。
- 因 item_id 變更後不再帶 `decodable_to`，天然防重複觸發；無需額外 flag。

手套線索筆記：

- 首次 `equip` 或首次 R 查看 `fingerless_gloves` → `add_knowledge(clue_gloves_decoder)` +「已記入筆記」提示。
- 一次性：以 `has_note("clue_gloves_decoder")` 判斷避免重覆；`has_knowledge()` 只用於身份 gate，不用於線索筆記去重。

不回掃背包：裝備手套**不**觸發背包既有可解碼物的批次轉換；先拿後戴情況由 R 查看補解。

待 content 補：兩物 id 命名定案、icon、解碼前/後 description、解碼 MessageBox 敘事全文、手套訊息字樣。

待確認實作細節：解碼能力旗標掛在裝備物 metadata 的具體欄位名與型別。

### 2-C 容器擴充（accepted_item / deposit_locked）

對應規格：`遊戲規格書.md > External Container API > 容器選項擴充` + `Phase 2 > 2-C 驗收意圖`。

GameState 改動（純資料層）：

- `configure_container(container_id, slot_count, accepted_item: Array[String] = [], deposit_locked: bool = false)`：新增兩個選用參數，預設值使既有呼叫不受影響。旗標隨容器狀態存放（與 slot 內容同一份 runtime state）。
- `get_container_config(container_id) -> Dictionary`：回傳 `slot_count` / `accepted_item` / `deposit_locked` 的 deep copy；UI 判斷失敗訊息時使用此 API。
- `move_one_item_to(target, instance_id)` 守衛新增兩條（在既有「目標可容納」檢查之外）：
  - 目標容器 `accepted_item` 非空且來源 item_id 不在清單 → `return false`（不變動來源/目標）。
  - 來源容器 `deposit_locked == true` → `return false`（不可作為搬運來源）。
- 回傳型別不變（bool）。不在 GameState 層回傳原因碼。移動成功時 emit `item_moved(move)`，供 2-B 等後續機制定位 target instance。

UI 層（接線於 2-D/2-E，2-C 僅資料層）：

- 雙欄移動前，UI 透過 `get_container_config()` 讀取目標容器 `accepted_item` / 來源容器 `deposit_locked` 預判，挑訊息：`放不下了。`（滿）/ `放不進去`（白名單拒）/ 取出被擋字樣（deposit_locked）。
- `move_one_item_to` 為權威 guard，UI 預判失誤也不會破壞資料。

不動：`get_container` / `seed_container` / `discard_item` 行為不變。`seed_container` 為 dev/預載入口，不受白名單約束（插槽不經 seed）。

待 content 補：deposit_locked「取出被擋」訊息字樣（2-E 定）。

待確認實作細節：旗標存放結構（與 container slots 同 dict 或分開 metadata dict）。

### 2-D 偵測終端（投影時鐘）+ 聲納 reveal

對應規格：`遊戲規格書.md > Phase 2 > 2-D 驗收意圖`；流程與文字見 `subdocs/地點/主角公寓.md`。

線索麵包屑道具（查看即消耗）：

- 新增 item `nutrition_bar_synth_blueberry`（合成藍莓口味營養棒，`category=consumable`，`max_stack` 待定），預載於 `fridge_storage`。
- icon 使用 `res://assets/generated/sprites/items/synthetic_blueberry_nutrition_bar/icon.png`。
- item metadata 新增欄位 `consume_grants_note`（String，note_id）。帶此欄位的物經 **E 使用 或 R 查看** 任一觸發：跳 MessageBox 顯示空包裝藏字敘述 → `add_knowledge`(該 note payload) → 移除該 instance（消耗）；R 路徑於 message 關閉後一併關 modal。兩路徑同效，仿手套解碼雙觸發。
- 一次性天然成立：instance 移除後不再存在，無需 flag。
- 移除 instance 沿用既有 discard/remove 容器路徑（在背包或外部容器皆可消耗）。

背包 E 使用動作（Phase 1-E footer 修正，2-D 連帶；對應規格書「物品操作」段更新）：

- 背包單獨（`I`）footer 的 E 段改為**依選中物品動態決定**（取代 Phase 1-E 固定字串）：
  - 可裝備物 → `E: 裝備`（未裝備）/ `E: 卸下`（已裝備）。
  - `category == "consumable"` 或標記可反覆使用 → `E: 使用`。
  - 其餘（無對應動作，如 `old_work_badge`）→ E 段不顯示。
- 按 E dispatch（單獨背包內）：
  - 可裝備 → 既有 equip / unequip。
  - 帶 `consume_grants_note` 的 consumable → 上述消耗 + 給筆記流程。
  - 無效果 consumable（如 `canned_food`）→ FloatingToast「現在用不上。」，不消耗。
  - 可反覆使用物 → 執行效果、不消耗（預見類型，暫無實例，不提前實作）。
- 雙欄左欄（背包欄）E 仍為「移動」，不變；`使用` 只在單獨背包出現。R 查看路徑於雙欄 / 單獨皆可，故營養棒在冰箱雙欄按 R 亦可消耗得線索。

待確認實作細節：「可裝備」「可反覆使用」如何由 item metadata 判定（category vs. 專屬 flag）；footer 動態字串組裝點；instance 消耗走 `discard_item` 或新 remove API。

投影時鐘互動物（= 偵測終端）：

- 沿用 `interactable_area.gd`；`required_knowledge` = 投影時鐘線索 note_id（取得線索前沿用既有 gate，不顯示 prompt / 不可互動）。
- 互動 E → 呼叫聲納啟動（見下）。

聲納狀態機（掛 `apartment_room.gd` 或抽獨立 helper node，二擇一待定）：

- 啟動 = 進入聲納 active 狀態 + 開 10 秒倒數計時器。
- 每 tick 依玩家 x 與隱藏插槽 x 的 1D 距離算強度 → 驅動短音播放頻率 + 視覺強度條；聲納短音資源為 `res://assets/sound/sonar_ping.wav`。
- 玩家在最強區帶寬內（`abs(dx) <= 36`）累積鎖定進度 4 秒（鎖定環）；離開帶寬則鎖定進度重置（容許帶寬內微動，不要求像素級不動）。prompt 文字「定位中…」。
- 鎖定環滿 → reveal：播放 `res://assets/sound/hidden_slot_reveal.wav`，顯示隱藏插槽為一般互動容器（`configure_container("apartment_slot", 1, ["decoder_cube"], true)`，見 2-C）；聲納狀態關閉、移除投影時鐘互動。
- 10 秒歸零且未鎖定 → 聲納關閉、不 reveal；投影時鐘仍可再互動重啟（狀態機可重入）。
- UIMode 進大型 UI（INVENTORY/NOTEBOOK 等）時暫停聲納 tick + 倒數 + 鎖定進度（不歸零）；回 NONE 恢復。
- reveal 後聲納不可再啟動（投影時鐘互動已移除，狀態機對重複啟動 idempotent）。

待 content 補：強度→短音播放頻率映射、10 秒是否可調、強度條視覺、鎖定環視覺、營養棒/線索/時鐘 prompt 與敘述文字、`nutrition_bar_synth_blueberry` max_stack。

待確認實作細節：聲納狀態機掛 `apartment_room.gd` 內或抽獨立 node；視覺強度條 UI 元件歸屬；鎖定環滿秒數與 10 秒窗的關係（dwell 須 ≤ window）；隱藏插槽 container_id 命名。

### 2-E 插槽放入整合 + MESSAGE/CONFIRM overlay 重構

對應規格：`遊戲規格書.md > Phase 2 > 2-E 驗收意圖` + `UIMode > overlay modal 的 caller 還原`。

UIMode overlay 重構（先做，放入鏈依賴它）：

- `MESSAGE` 改為與 `CONFIRM` 同類的 **overlay modal**：進入記住 caller mode、不關 caller UI、退出還原焦點。
- 既有 `enter_confirm` / `exit_confirm` 一般化為 `enter_overlay(mode)` / `exit_overlay()`，`MESSAGE` / `CONFIRM` 共用 `_caller_mode`。
- caller UI 在 overlay 期間凍結接鍵（不吃方向 / E / R / T）；overlay 優先吃鍵、推進文字、`Esc` 關閉。
- 遷移既有呼叫：examine(2-A) / 解碼(2-B) 開 MessageBox 改走 `enter_overlay(MESSAGE)`（caller 多為 NONE / INVENTORY），視覺行為不變。
- MVP 單層 overlay（不疊 MESSAGE on CONFIRM）。
- 回歸：1-B~1-E 大型 UI 互切、CONFIRM 丟棄確認、focus 還原全測。

放入觸發鏈（`apartment_room.gd`）：

- listen `container_changed("apartment_slot")` → 若 `decoder_cube` 在槽內 且 未 `has_knowledge("identity_door_unlock_method")` → 跑鏈。
- 鏈：播電磁聲響（音效待定）→ `enter_overlay(MESSAGE)` 顯示語音 MessageBox（疊在雙欄上）→ 關閉（`exit_overlay`）後 `add_knowledge(identity_door_unlock_method payload)`。
- 一次性守衛：因 `add_knowledge` 延後到 MessageBox 關閉才寫，逐字顯示窗內 `has_knowledge` 仍為 false，需另設 transient flag。`apartment_room.gd` 持 `_slot_unlock_sequence_started: bool`；觸發條件為 `!_slot_unlock_sequence_started and !has_knowledge(...)`，進鏈第一步即 `_slot_unlock_sequence_started = true`，防同窗重入。`deposit_locked` 另使該槽不再變動，`container_changed` 長期亦不再為此槽觸發。
- `add_knowledge` 時機：MessageBox 關閉（`exit_overlay`）之後寫入。
- 完成後插槽常駐、方塊鎖在格內（deposit_locked），雙欄可開看、不可取出。
- 語音 / 筆記全文見 `subdocs/地點/主角公寓.md > 互動物：隱藏插槽`。

大門（B9，不改現有程式）：

- 沿用 `DoorExitArea` 的 `has_knowledge("identity_door_unlock_method")` gate；MVP 顯示 `door_opened`，不轉場。

待 content 補：電磁聲響音效素材。

待確認實作細節：聲響與 MessageBox 的先後 / 同步時序；overlay 是否需 `get_base_mode()` 之類查詢（MVP 用 `_caller_mode` 即足）；遷移 enter_confirm 既有 caller 的相容性。
